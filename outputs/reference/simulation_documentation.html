Simulation Documentation

Overview

This project implements a discrete-event simulation of a clinic appointment booking system. The model is a day-level appointment model with two patient classes and first-come-first-served booking.

The simulation includes:

  • class-specific daily Poisson arrivals
  • daily random permutation of arrivals
  • same-day booking allowed
  • delay-dependent balking
  • delay-dependent no-show
  • start-of-day cancellations for future appointments only
  • no same-day cancellations
  • no rebooking of no-show slots
  • a rolling day-level booking calendar
  • burn-in, measurement, and cooldown periods

The simulation is designed to evaluate how appointment availability, patient behavior, and clinic capacity affect access, booking delay, service completion, and utilization.

Time and State

Time is modeled one day at a time.

Let:

  • D = current day
  • S = number of appointment slots available per day
  • H = booking horizon in days
  • r ∈ {0, ..., H-1} = residual day offset, where r = 0 means today

The main state is a rolling day-level calendar. For each residual day r, the model stores the patients currently scheduled for day D + r.

The model distinguishes between two concepts:

  • tau: the original offered booking delay in days
  • r: the current remaining days until the appointment

This distinction matters because balking is based on the offered delay tau. For patients who accept and enter the calendar, no-show behavior is also based on the original booking delay tau. The calendar itself is organized by current residual day r.

For example, a patient shown in the calendar at r = 0 with tau = 9 is scheduled for today, but originally accepted an appointment that was 9 days away.

Patient Classes

Each patient class has its own parameters:

  • lambda_per_day: mean daily arrival rate
  • balk_prob: delay-dependent probability of rejecting an offered appointment
  • cancel_prob: probability of canceling a future appointment
  • no_show_prob: delay-dependent probability of not attending a scheduled appointment
  • value: value generated by a completed visit

Daily arrivals are generated separately for each class. The model then pools all arrivals for the day and randomly permutes them into one FCFS arrival sequence.

Probability Rules

The current implementation supports threshold probability rules.

A threshold rule has the form:

\[ p(\tau)= \begin{cases} \text{low}, & \tau \le \text{threshold} \\ \text{high}, & \tau > \text{threshold} \end{cases} \]

where tau is the offered or accepted booking delay.

For example, if threshold = 7, then:

  • tau = 0, 1, ..., 7 uses the low probability
  • tau = 8, 9, ... uses the high probability

If the booking horizon is 10 days, possible offered delays are 0 through 9. Therefore, setting threshold = 9 means the high probability is never used.

Daily Simulation Order

For each day, the simulation follows this order:

  1. Apply cancellations
    Future appointments only (r >= 1) may cancel with class-specific probability. Same-day cancellations are not allowed.

  2. Record the start-of-day summary state
    The model records the summary state after cancellations and before new arrivals are processed.

  3. Generate daily arrivals
    For each class i, daily arrivals are generated as a Poisson random variable with mean lambda_per_day.

  4. Randomly permute arrivals
    All arriving patients for that day are pooled and randomly permuted to determine FCFS order.

  5. Offer appointments in FCFS order
    Each patient is offered the earliest available day in the booking horizon, including same-day availability (r = 0) if capacity remains.

  6. Record offered delay
    If a valid slot is found, the offered delay tau is recorded for the offered-delay metric. This happens before the balking decision, so patients who later balk are included in the offered-delay calculation.

  7. Apply balking
    If a patient is offered delay tau, they reject the offer with probability b_i(tau). If they balk, they do not enter the appointment calendar.

  8. Book accepted patients
    If the patient does not balk, they are booked into the offered day. Their original booking delay tau is stored with the booking.

  9. Capacity exhaustion / no offer
    If the booking horizon is full, the current patient and all remaining arrivals for that day are counted as no_offer.

  10. Capture final calendar snapshot
    On the final simulated day, the model captures the final calendar view after arrivals are processed but before service and before the calendar rolls forward.

  11. Resolve today’s appointments
    Patients scheduled for today either attend or no-show. No-show slots are not rebooked.

  12. Compute utilization and advance the calendar
    Daily utilization is computed from the number of patients served that day divided by the number of slots per day. The model then removes the current day, shifts future days forward, and appends a new empty future day.

Burn-In, Measurement, and Cooldown

The simulation separates time into three phases:

  • Burn-in: allows the system to move away from the artificial empty starting state
  • Measurement: the period during which primary outputs are recorded
  • Cooldown: allows patients entering during measurement to complete their appointment process

This design makes the reported outputs less sensitive to the empty-calendar initial condition.

Class-level patient metrics are tracked for patients who arrive during the measurement period. These include arrivals, bookings, balks, no-offer counts, cancellations, no-shows, and completed service.

Slot-level utilization metrics are recorded for service days that fall within the measurement period.

Summary State

The summary state is a compact representation of the booking calendar by patient class and residual day.

It is defined as:

\[ X^D_{i,r} = \text{number of class-}i\text{ patients scheduled for day }D+r \]

where:

  • i = patient class
  • r = residual day offset
  • D = current day

With two classes and a 10-day horizon, a summary state has the form:

{
    1: [count at r=0, count at r=1, ..., count at r=9],
    2: [count at r=0, count at r=1, ..., count at r=9],
}

The summary state is recorded after start-of-day cancellations and before new daily arrivals are processed. It is therefore a start-of-day calendar occupancy snapshot, not a daily performance summary.

The summary state does not directly record:

  • arrivals
  • balking
  • no-offer counts
  • no-shows
  • served patients
  • utilization
  • total value

Those outcomes are recorded separately in the simulation metrics.

Final Calendar View

The final calendar view is a patient-level diagnostic view of the booking calendar.

Each row corresponds to one residual day r, and each entry is either:

  • (class_id, tau) for a booked patient
  • 0 for unused capacity

For example:

r=0: [(1, 9), (2, 8), (1, 3), 0, 0]

means that today’s row contains three booked patients and two unused slots.

The tuple (1, 9) means:

  • patient class = 1
  • original booking delay tau = 9

It does not mean that the patient currently has 9 days remaining. If the patient is in row r = 0, the appointment is scheduled for today.

Metric Definitions

The simulation reports class-level metrics, slot-level diagnostic metrics, and aggregate system-level metrics.

Class-Level Metrics

Arrivals

arrivals counts the number of patients from a given class who arrive during the measurement period.

Booked

booked counts patients who received an offer and accepted it. These patients enter the appointment calendar. A booked patient may later cancel, no-show, or be served.

Balked

balked counts patients who received an offered slot but rejected it. In the threshold-rule setting, the balking probability depends on the offered delay tau.

Offered

offered is defined as:

offered = booked + balked

This is the number of patients who received a valid slot offer. Patients counted as no_offer are excluded because no appointment delay was available for them.

No Offer

no_offer counts patients who could not be offered any slot because the booking horizon was full.

In the current engine, if the horizon is full for one patient, that patient and all remaining arrivals for that day are counted as no_offer.

Canceled

canceled counts tracked future appointments that cancel before the service day. Same-day cancellations are not allowed in the current model.

No-Show

no_show counts booked patients who reach their appointment day but do not attend. No-show slots are not rebooked.

Served

served counts patients who attend and complete service.

Mean Accepted Booking Delay

The accepted booking delay is the average original booking delay among patients who accepted an appointment:

\[ \text{mean accepted booking delay}_i = \frac{ \sum_{\text{accepted patients of class } i} \tau }{ \#\text{accepted patients of class } i } \]

This excludes patients who balked and patients who received no offer.

Mean Offered Booking Delay

The offered booking delay includes both patients who accepted and patients who balked:

\[ \text{mean offered booking delay}_i = \frac{ \sum_{\text{offered patients of class } i} \tau }{ \#\text{offered patients of class } i } \]

where:

\[ \#\text{offered patients}_i = \#\text{booked patients}_i + \#\text{balked patients}_i \]

Patients with no_offer are excluded because they did not receive a valid appointment delay.

This metric is useful because accepted booking delay can be biased downward when patients with long offered delays are more likely to balk.

Percent Serviced

For each class:

\[ \text{percent serviced}_i = \frac{ \text{served}_i }{ \text{arrivals}_i } \]

This measures the share of demand from class (i) that ultimately becomes completed service.

Slot-Level Metrics

Slot-level metrics are measured at service time for r = 0.

Booked Slots

booked_slots counts how many same-day service slots were booked during the measurement period. This includes patients who later no-show.

This metric is diagnostic. It measures scheduled capacity, not completed service.

Served Slots

served_slots counts how many measured service slots resulted in completed visits. This is the slot-level count that drives utilization.

No-Show Slots

no_show_slots counts booked measured service slots that were lost because the patient did not attend.

For measured service days:

booked_slots = served_slots + no_show_slots

when all booked same-day appointments either attend or no-show.

Aggregate Metrics

Average Utilization

Average utilization is computed as the average daily share of slots that become completed visits:

\[ \text{daily utilization}_d = \frac{ \text{served patients on day } d }{ \text{slots per day} } \]

\[ \text{average utilization} = \frac{1}{M} \sum_{d=1}^{M} \text{daily utilization}_d \]

where (M) is the number of measured days.

This utilization definition excludes:

  • no-shows
  • balked patients
  • no-offer patients
  • unbooked slots

In other words, utilization measures the share of clinic capacity that became completed visits.

Overall Percent Serviced

\[ \text{overall percent serviced} = \frac{ \sum_i \text{served}_i }{ \sum_i \text{arrivals}_i } \]

This measures the aggregate share of measured arrivals who are ultimately served.

Total Served

total_served is the total number of completed visits across all classes:

\[ \text{total served} = \sum_i \text{served}_i \]

Total Value

total_value is the sum of each served patient’s class-specific value:

\[ \text{total value} = \sum_{\text{served patients}} \text{value}_i \]

where (i) is the patient’s class.

Repository Structure

  • simulation/model.py: parameter objects, booking objects, metrics, and results
  • simulation/engine.py: simulation engine

  • scripts/run_simulation.py: runs one scenario
  • scripts/compare_scenarios.py: compares two scenarios
  • scripts/generate_metric_analysis_figures.py: builds report figures and manifest
  • experiments/: parameter sweeps used by reports
  • configs/: scenario parameter files
  • notebooks/: exploratory notebooks

Running the Simulation

Install dependencies:

pip install -r requirements.txt

Run the baseline scenario:

python run_simulation.py

Run the scenario comparison script:

python compare_scenarios.py
